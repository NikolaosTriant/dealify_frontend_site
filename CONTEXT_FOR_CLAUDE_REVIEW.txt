==================================================
1) PRODUCT SUMMARY
==================================================
TrustLayer is a real-estate transaction orchestration UI for the pre-notary phase.
Primary pilot audience: RE/MAX Greece brokers.

Core concept:
- Broker coordinates parties (lawyer, engineer, notary, accountant, etc.)
- Deals move through template-based stages/tasks with dependencies
- Client gets link-based portal (no login)
- Members get assigned task-focused views

Current implementation status in this repo:
- Frontend-only demo/prototype
- Mostly mock data in components
- No backend/API integration yet
- No real auth yet

5) ROUTING / NAVIGATION MODEL
==================================================
Routing is custom path-state mapping (not full react-router setup), implemented in:
- src/app/screens.ts
- src/app/App.tsx

Supported paths:
- /dashboard -> broker-dashboard
- /listings -> broker-listings
- /leads -> broker-leads-matching
- /transaction -> broker-transaction-detail
- /transaction/:id/documents -> broker-document-review
- /member-tasks -> broker-member-task-view
- /templates -> broker-template-builder
- /client/:token -> broker-client-portal
- /calculator -> public-calculator

Navigation behavior:
- Uses history.pushState + popstate listener
- Active screen held in App state
- transactionId state used for document review path

6.8 UI Πελατών -> Πύλη Πελάτη (/client/:token)
Main file:
- src/app/components/ClientPortalPage.tsx

This is a mobile-first composite client experience with two phases.

Phase A: Document Submission (initial)
Implemented:
- Top property card (title/address/price/photo placeholder)
- Required docs list from mock template
- Per doc row:
  - name + description
  - status badge (Pending/Uploaded/Approved/Rejected)
  - upload input (PDF/JPG/PNG, max 10MB)
  - rejection comment display
  - filename/date display
- row color states:
  - approved green
  - rejected red
- progress bar X/Y submitted
- auto transition trigger when all approved

Demo behavior note:
- Uploaded docs are auto-approved after short timeout to simulate broker review.

Phase B: Process Tracker / simplified skill tree
Implemented:
- completion ring (% based on completed tasks)
- vertical stepper/stages with role-only visibility
- statuses:
  - completed with date
  - in progress
  - locked
  - overdue (red)
- expanded stage tasks
- Send Reminder button for overdue/in-progress tasks

Nudge/reminder behavior implemented:
- confirmation modal: "Send reminder to [role] via email?"
- on send:
  - task stores reminderSentAt timestamp
  - button disabled for 48h cooldown
  - label changes to "Reminder sent [date]"
  - local notification record added with:
    - type = nudge
    - channel = email
    - brokerNotified = true

Activity feed section:
- collapsible Activity Log
- reverse chronology entries
- icon/title/description/timestamp
- tone styles:
  - info (blue)
  - success (green)
  - alert/action-needed (red)
- filters: All / Action Needed / Milestones

Contact & Help section:
- broker contact card with tel/mailto links
- FAQ accordion with requested questions
- message form (subject + textarea + send)
- send appends mock activity event

Bottom nav tabs in process phase:
- Διαδικασία
- Activity
- Βοήθεια

Current limitations:
- Token not validated
- No real file storage
- No real email/SMS/notification backend

==================================================
7) DATA / STATE CHARACTERISTICS (IMPORTANT)
==================================================
Current data model is component-local mock state.
There is no shared global store and no backend sync.

Implications:
- Refresh resets most state
- Cross-screen actions (e.g. Add to Leads from Listings, Start Deal from New Leads) are mostly demo placeholders
- Needed next step for production-quality demo is at least a temporary central mock-store (context/zustand) or API layer

8) KNOWN GAPS / RISKS FOR REVIEW
==================================================
Functional gaps:
- No authentication/authorization
- No backend API integration
- No database persistence
- No real notification channels (email/SMS)
- No document storage backend
- No true token lifecycle/security
- No role-based data access enforcement

UX/Product gaps:
- Some English labels remain in mixed Greek UI
- Some alerts are placeholder popups
- Client portal is currently reachable from broker shell for design convenience (not strict app separation)
- No full responsive QA pass across all broker pages yet

Engineering gaps:
- Custom router state works but is limited vs full react-router
- No tests (unit/integration/e2e)
- No type-safe API contracts yet
- No error boundaries / robust empty/loading states across all pages
